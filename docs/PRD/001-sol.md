# PRD 001: Sol

**Created:** 2026-02-28
**Updated:** 2026-02-28
**Status:** Active

## 1. Vision

Sol is a mobile-first web application that serves as a full remote frontend to pi (pi.dev), a CLI-based AI coding agent. It runs on the developer's laptop and is accessed from an iPhone via Safari over Tailscale VPN.

Sol enables complete pi-based workflows while away from the keyboard: browsing sessions, reading conversation history, sending prompts, reviewing agent-generated code, inspecting modified files, switching models, and starting new sessions. It replaces the need for a mobile SSH terminal by providing a touch-optimized, polished interface purpose-built for working with pi on a phone.

Named after Sol Robeson, Max's mentor in the film π.

## 2. User Stories

### 2.1 Session Management

- As a developer, I want to see all my pi sessions grouped by project directory so I can find past conversations without remembering file paths.
- As a developer, I want sessions sorted by recency so the most relevant ones appear first.
- As a developer, I want to see session metadata (message count, timestamps, first message preview) so I can identify sessions at a glance.
- As a developer, I want to connect to an active pi session so I can continue working on it from my phone.

### 2.2 Conversation

- As a developer, I want to view a session's full conversation history with proper formatting so I can review what the agent did.
- As a developer, I want user messages, assistant responses, and tool results visually distinguished so the conversation is easy to follow.
- As a developer, I want to expand and collapse tool calls so I can focus on the conversational flow without clutter.
- As a developer, I want to see model info, token counts, and cost per message so I can understand resource usage.
- As a developer, I want to send prompts to the active session with a mobile-friendly input so I can direct the agent from my phone.
- As a developer, I want to see streaming agent responses in real-time so I know what the agent is doing as it works.

### 2.3 Tree Navigation

- As a developer, I want to visualize the conversation tree (branches, forks, compactions) so I can navigate alternate paths the agent explored.
- As a developer, I want to tap a branch point and switch to viewing a different branch so I can compare approaches.

### 2.4 File Inspector

- As a developer, I want to see files the agent has modified or created (git working tree changes) so I can review its work.
- As a developer, I want to view full file contents with syntax highlighting so I can read agent-generated code on my phone.
- As a developer, I want to view markdown files with a toggle between rendered and raw so I can read documentation the agent wrote.
- As a developer, I want to view git diffs for modified files so I can understand what changed.

### 2.5 Model Management

- As a developer, I want to switch the active model mid-session so I can adjust cost/capability tradeoffs from my phone.

### 2.6 Search

- As a developer, I want to search across all sessions by keyword so I can find specific conversations or topics.
- As a developer, I want search results to link directly to the matching session and entry so I can jump to context quickly.

### 2.7 Agent Autonomy & Testing

- As a developer, I want agents to verify their own implementations so that ticket completion doesn't require manual human testing.
- As a developer, I want automated regression detection so that new features don't silently break existing ones.
- As a developer, I want every ticket to include runnable acceptance tests so agents have a clear, machine-verifiable definition of done.

### 2.8 Mobile Experience

- As a developer, I want the UI to feel like a polished native app with snappy navigation so working from my phone doesn't feel like a compromise.
- As a developer, I want a dark theme with a strong design system so the app is comfortable to read and visually consistent.

## 3. Functional Requirements

### 3.1 Session List (MVP)

- Discover all sessions via `SessionManager.listAll()` from the pi SDK.
- Group sessions by project directory.
- Sort by last modified descending within each group.
- Display: session name or first message preview, message count, created/modified timestamps.
- Indicate which sessions have an active pi RPC subprocess managed by Sol.

### 3.2 Conversation View (MVP)

- For historical sessions: load entries via `SessionManager.open()` and render the current branch.
- For active sessions: retrieve messages via pi RPC `get_messages` command.
- Render the current branch (root to leaf) as a scrollable chat view.
- Handle all message types: user, assistant, toolResult, bashExecution, custom, branchSummary, compactionSummary.
- Display assistant metadata: model, provider, token usage, cost.
- Tool calls and results are collapsed by default, expandable on tap.
- Optimized for reading conversational flow between user and assistant.

### 3.3 Prompt Input (MVP)

- Mobile-friendly text input with iOS autocorrect and dictation support.
- Send prompts to active session via pi RPC `prompt` command.
- Visual feedback for send state (sending, sent, error).
- Support for steering (`steer`) and follow-up (`follow_up`) RPC delivery modes.

### 3.4 Streaming Output (MVP)

- Sol spawns `pi --mode rpc` subprocesses for active sessions.
- Pi RPC `message_update` events (JSON lines on stdout) are forwarded to the browser via SSE.
- Tokens rendered as they arrive with <50ms perceived latency.
- Tool execution status visible via `tool_execution_start/update/end` events.

### 3.5 File Inspector (MVP)

- List files from git working tree changes (modified, added, untracked) via `git status` in the session's `cwd`.
- View full file contents with syntax highlighting (Shiki) via direct `fs` read.
- View markdown files with toggle between rendered and raw. Rendered view uses typographic prose styling (heading hierarchy, list indentation, blockquote accents, code block formatting, table borders) tuned to dark theme design tokens for comfortable mobile reading.
- View git diffs in line-by-line mode (diff2html) via `git diff` in the session's `cwd`.

### 3.6 Model Switching (MVP)

- List available models via pi RPC `get_available_models` command.
- Switch model via pi RPC `set_model` command.

### 3.7 Tree Navigation (MVP)

- For historical sessions: build tree via `SessionManager.open()` and `sm.getTree()`.
- For active sessions: retrieve entries via RPC `get_messages` and build tree client-side, or read the session file directly (pi appends are safe to read concurrently).
- Identify branch points (entries with multiple children).
- Breadcrumb-style navigation to switch between branches.
- Show compaction and branch summary entries at their tree positions.

### 3.8 Search (MVP)

- Full-text search across all sessions using `allMessagesText` from `SessionInfo` for fast pre-filtering.
- Within-session search scanning individual entries server-side.
- Return matching sessions with hit count and preview snippets.

### 3.9 Testing & Agent Verification (MVP)

- Vitest as the sole test runner, configured for ESM + TypeScript strict mode.
- Unit tests for all backend modules containing extractable logic.
- API integration tests for every Express endpoint using `supertest`.
- Co-located test files: `src/foo.test.ts` alongside `src/foo.ts`.
- Shared test utilities and mock factories in `src/__test__/`.
- NPM scripts: `npm test` (run all), `npm run test:watch`, `npm run test:coverage`.
- Every implementation ticket requires tests that verify its acceptance criteria.
- Agents must run `npm run build && npm test` before marking any ticket complete.

### 3.10 API

Sol's Express server exposes REST + SSE endpoints. Active session endpoints proxy to `pi --mode rpc` subprocesses. Historical and file endpoints use the SDK and filesystem directly.

See PRD Section 3.10 for endpoint table.

| Endpoint | Method | Backend | Description |
|----------|--------|---------|-------------|
| `/api/sessions` | GET | SDK `SessionManager.listAll()` | Session list with metadata, grouped by project |
| `/api/session/:id` | GET | SDK `SessionManager.open()` | Full entries for one session (historical) |
| `/api/session/:id/connect` | POST | Spawn `pi --mode rpc` | Attach to session, returns RPC process handle |
| `/api/session/:id/stream` | GET (SSE) | RPC stdout → SSE | Stream agent events for active session |
| `/api/session/:id/prompt` | POST | RPC `prompt` command | Send prompt to active session |
| `/api/session/:id/steer` | POST | RPC `steer` command | Steer active session |
| `/api/session/:id/abort` | POST | RPC `abort` command | Abort current operation |
| `/api/session/:id/model` | PUT | RPC `set_model` command | Switch model for active session |
| `/api/session/:id/models` | GET | RPC `get_available_models` | List available models |
| `/api/session/:id/state` | GET | RPC `get_state` | Session state (streaming, model, etc.) |
| `/api/session/:id/commands` | GET | RPC `get_commands` | Available slash commands and skills |
| `/api/tree/:id` | GET | SDK `sm.getTree()` | Tree structure with children |
| `/api/files/:id` | GET | `git status` via `child_process` | Git working tree changes for session's cwd |
| `/api/files/:id/content` | GET | `fs.readFile` | Full file contents |
| `/api/files/:id/diff` | GET | `git diff` via `child_process` | Git diff for a file |
| `/api/sessions/search` | GET | SDK `SessionManager.listAll()` + filter | Search across sessions |
| `/api/session/:id/search` | GET | SDK `SessionManager.open()` + scan | Search within a session |

## 4. Non-Functional Requirements

### 4.1 Performance

- Streaming latency: <50ms time-to-first-token visible on screen after the server emits it.
- Navigation between views: <100ms transition, no perceptible jank.
- Session list load: <200ms for ~50 sessions.
- Production JS bundle: <60KB min+gz (excluding lazy-loaded Shiki WASM).

### 4.2 Security

- No application-level authentication. Tailscale provides network-level security.
- Server binds to `0.0.0.0` for Tailscale access.

### 4.3 Design

- Dark theme with WCAG AA contrast ratios (4.5:1 body text, 3:1 UI components).
- Minimum touch targets: 44×44pt (Apple HIG).
- Consistent spacing on 4px base grid.
- Mobile-first responsive design; primary target is iPhone Safari.
- The app should feel like a polished native app, not a developer tool.

### 4.4 Technology

- Frontend: Preact + Tailwind CSS v4 + Headless UI, built with Vite.
- Backend: Express + TypeScript, run via `tsx`.
- Active session integration: `pi --mode rpc` subprocesses managed by Sol.
- Session discovery: `@mariozechner/pi-coding-agent` SDK (`SessionManager`).
- Syntax highlighting: Shiki (lazy-loaded grammars).
- Markdown rendering: marked.
- Diff rendering: diff2html (line-by-line mode).
- Streaming: SSE (Server-Sent Events), bridging pi RPC stdout to browser.
- Server port: 8081.

See [Frontend Technology Prestudy](../prestudy-frontend-technology.md) for frontend rationale.
See [ADR 005: Pi RPC Integration](../ADR/005-pi-rpc-integration.md) for active session architecture.
